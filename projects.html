Retail Point of Sale System

PreviousNext

Retail Point of Sale System is a Java Application that utilizes the use of GUI and DBMS, it is one of the final requirement for our subject Application Development and Emerging Technologies.

for the Graphical User Interface, I used Window Builder because the deadline is cut short to 1 week and the rubric has a huge percentage for the DBMS utilization. Moreover, it is focused on the backend conflicts and errors. 

The first picture depicts the Home Page, where there are 4 buttons to choose from. The User can sell item, check the inventory, modify the inventory, either edit, add new data, or delete data from the database, and refund.

The second picture is the Sell Window. the blue textfields indicate that if a valid input is entered, the gray textfields, which is uneditable, would be automatically filled by a relative value to the entered input in the blue text fields, there are four buttons, the new, which renews the sell window and removes all the values entered and implied on the table, back to home, which is easily the most common out of the 4 buttons that takes us back to the home page, the Bill, which sums up all the prices and the payments, and finally, the print button that searches for the hardware's printer in order to directly print what is in the text are which is the receipt.

The third picture is the Inventory Window, which basically just shows the content of the database MSAccess file, which is the id, quantity, price, and the name of the item.

the fourth picture is the Modify Window, it has the most capability to manipulate the database of all windows, the blue textfields are still indicators of the gray textfields if a valid input is entered, item id was made to autogenerate when an existing id is already vacating an item when adding a new item, I certainly put a warning window which has a Yes and No choice in order to delay the decision of the user for it would be a bit crucial for the Database if changes were made.

the last picture is the Refund Window, customers may refund their bought item if the user did a product checking and their customer id matched what's in the system. The customer ID is also autogenerated by a logic in the system, therefor, removing the problem of having identical id's that would create problems for refunding. When refunds are made, the item quantity reverts the refunded items to its original count.

Voting System

PreviousNext

Voting System is a project that was made by me, Famke Francisco, and Edmond Valdez. It was our final requirement for the subject Object Oriented Programming back in 2019 as a second year student. It has a simple instruction of making a Graphical User Interface that can utilize the Insert, Update, and Delete functions of the database.

It has a basic GUI because it was introduced to us not long ago before starting the project. The database that was used is an MSAccess file.

The first picture includes a Student Login where it is programmed to accept students who has not voted yet. The Second Picture is the Voting Window, which has a categories and a dropdown on who to vote into their respective positions.

The third picture, on the other hand, is Admin Login, where the voting administrator logs in, he/she has more capability to control the data that has been inside the database. The fourth picture is the Administrator Window, it depicts the 5 buttons that the admin authorize to control the flow of data. The fifth picture is the Add Candidate, where the admin has the capability of registering a new candidate in the database.

The sixth picture is Add Voter Window and the Delete Candidate Window, where the admin can authorize a student to register to vote and to be included in the database, and remove a Student Candidate if he/she withdraws. The seventh picture is the Update Candidate Window, it is the window and admin capability to update a student's details by entering a student ID. The last picture is the Show Data Window, where there are buttons of certain positions, and by clicking them, shows the table of the name of the candidates and the tally of their votes.

PHUSE

PreviousNext

Please Help Us Shop Easily, or PHUSE for short, is a Mobile Application that is produced for our thesis requirement. It is a Mobile Application that gets grocery lists as inputs from the user then utilizes A Star Algorithm with Ant Colony Optimization to seek for the most optimized route and time. 

The initiation for the platform where the application would be produced is React Native. It is an open-source JavaScript framework that is designed to build mostly for mobile application. 

We created a draft of how the flow for our mobile application would be (See picture 1) and our attempt to create a modal after creating a list after the loading screen as initiation (See picture 2). The modal for the list were finalized after creating the modal header and components that were conceptualized at the draft(See picture 3). The cart screen was also made after creating the interaction for the grocery list button (See picture 4).

We also tried to utilize React Redux for our application in order to bring the data to the Redux store in order to dispatch all the actions and update the data from the item screen to the cart screen, but failed to do so. We shifted our Mobile Application Development platform to Unity after failing many attempts to Integrate React Native to Android Studio where A Star and Ant Colony would be used. The project was finished in Unity but left unfinished in using React Native.

We used an outdated compiler namely Turbo C++ in order to understand the fundamentals of programming further by exploring a language that has no automatic memory management.

The photos above are the final requirement for the subject Programming 1.  It is a simple logic based program that has several nested if else, do-while,  case statements, and multidimension array. (See picture 2).  Input score is assigning data to a two dimensional array indicating which team, player and how much score that specific individual did.  The exit trigger for the input score would be giving 0 as the input  (See Picture 3). If the choice in the menu would be to exit, it would display an exit message (See picture 4).

 Programming 2 introduced us to a more advanced C knowledge such as file handling and the use of structures, but the final requirement for Programming 2 is a batch-synchronous final practical examination, the reason why there is no documentation for it.
